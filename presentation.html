Understanding Agentic Search

Traditional Search: End-user types query → System searches → Results

Agentic Search: User asks LLM → LLM formulates optimized search queries → System searches → LLM synthesizes answer

The Key Insight

The LLM already performs semantic-to-keyword translation!

When user asks: "How do Windows and Linux handle TCP/IP differently?"

LLM generates: "Windows Linux TCP socket differences Winsock closesocket fork CreateProcess"

This is already the perfect BM25 query. Vector search adds nothing.

Why LLMs Make Vector Search Redundant

What LLMs Do Automatically:

  • ✓ Keyword extraction - Identify salient technical terms
  • ✓ Terminology precision - Use exact API names, not paraphrases
  • ✓ Query reformulation - Retry with different terms if needed
  • ✓ Multi-query strategies - Issue parallel queries from different angles

Vector search bridges semantic gaps:

"car repair" → "automotive maintenance"

BUT: With LLM agents, the LLM already bridges these gaps before calling search!

Why BM25 Dominates for Technical Documentation

1. Precise Terminology Matching

Technical docs use exact terms that appear in queries:

  • Winsock, closesocket, WSAStartup (Windows)
  • fork(), close(), POSIX (Linux)
  • These are not synonyms - they're literal API names

2. No Semantic Ambiguity

  • fork() doesn't mean "split" - it's a system call
  • socket has one technical meaning
  • Precision matters more than semantic similarity

When Vector Search Actually Adds Value

Vector search provides benefits when:

  • ✓ Natural language from end-users (not LLM-mediated)
  • ✓ Inconsistent terminology across documents
  • ✓ Cross-lingual search requirements
  • ✓ Conceptual similarity over keyword matching
  • ✓ Large, diverse corpus (1000s+ documents with varied terminology)

None of these apply to agentic search over technical reference documentation.
